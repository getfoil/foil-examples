Overview

Architecture & Trace Structure

Every chat session produces one Foil trace. Inside that trace, each user message creates an LLM span. When the model calls tools (customer lookup, order search, etc.), those become child spans nested inside the LLM span.

What a typical trace looks like in your Foil dashboard
Trace: customer-support-web (full session)
LLM: gpt-4o-mini (turn 1 — "Hi, my email is sarah@example.com")
Tool: lookup_customer
LLM: gpt-4o-mini (turn 2 — "What's the status of my keyboard order?")
Tool: lookup_customer
Tool: get_customer_orders
SDK (your app) Foil Platform =============== ============= new Foil({ apiKey }) -------> Authenticates via API key Identifies agent "customer-support-agent" foil.trace(callback) -------> Opens a new trace | (one per conversation session) | +-- ctx.startSpan() -------> Opens an LLM span inside the trace | | Records: model, input, config | | | +-- ctx.executeTools() -> Auto-creates child spans for each tool | | Records: tool name, args, result | | | +-- span.end() -------> Closes the span | Records: output text, token counts, errors | +-- (trace callback returns) -> Closes the trace, flushes all data foil.shutdown() -------> Flushes remaining data, closes connections
Step 1

Initialize the Foil SDK

The very first thing the app does is create a Foil client. This sets up the connection to Foil's platform and identifies which agent is being traced.

src/web.js lines 6, 18-24 
const { Foil } = require('@getfoil/foil-js');

const foilApiKey = process.env.FOIL_API_KEY;

const foil = new Foil({
  apiKey: foilApiKey,       // Your Foil API key (sk_live_...)
  agentName: 'customer-support-agent',  // Identifies this agent in your dashboard
  debug: !!process.env.FOIL_DEBUG,     // Set FOIL_DEBUG=1 to see SDK logs
});
i
What this does

Creates a Foil client that all tracing goes through. The agentName groups all traces under one agent in your dashboard, so you can set up evaluations, alerts, and profiles specifically for this agent. No HTTP calls happen yet — the SDK is just configured and ready.

+
Graceful degradation

If FOIL_API_KEY is not set, the app still works — it creates a mock context with no-op methods. You can develop locally without Foil, then enable tracing by setting the env var.

Step 2

Start a Trace (One Per Session)

When a user starts chatting, the app opens a Foil trace. A trace is the top-level container — think of it as "one complete conversation." Everything that happens in the session (LLM calls, tool uses, errors) is nested inside this trace.

src/web.js lines 98-127 (simplified) 
function getOrCreateSession(sessionId) {
  // Create a promise we control — resolving it ends the trace
  let resolveTrace;
  const pending = new Promise((resolve) => {
    resolveTrace = resolve;
  });

  const session = { ctx: null, end: null };

  // Start the trace — callback receives `ctx` and stays open
  // until we call resolveTrace()
  session.ready = foil.trace(
    (ctx) => {             // ctx = the tracing context
      session.ctx = ctx;   // Save it so HTTP handlers can use it
      return pending;      // Keep trace alive until session ends
    },
    {
      name: 'customer-support-web',  // Trace name (shown in dashboard)
      sessionId,                      // Links all turns to this session
      timeout: 0,                    // No auto-timeout — lives until ended
    },
  );

  session.end = () => {
    resolveTrace();        // This closes the trace and flushes data
    sessions.delete(sessionId);
  };

  return session;
}
i
What this does

foil.trace(callback, options) opens a trace and gives you a ctx (context) object. The trace stays open as long as the callback's promise is pending. In a web app, the session can last minutes or hours, so we use a manually-controlled promise: the trace stays alive across many HTTP requests, and only closes when the user leaves or clicks "New Session."

!
Key concept: ctx is your tracing handle

The ctx object is what you use for everything — starting spans, executing tools, recording signals. It's scoped to this one trace. Every method you call on ctx creates data nested inside this trace.

Step 3

Open a Span for Each LLM Turn

When a user sends a message, the agent opens an LLM span. This span captures: the user's input, which model was used, config (temperature, max tokens), and eventually the output and token counts. It stays open through the whole tool-calling loop.

src/agents/support.js lines 106-116 
// Build span input — use SDK content blocks for image tracing
const spanInput = isImage
  ? content(textPart || 'Customer attached this image:', ContentBlock.file(filePath))
  : userMessage;

// One LLM span per conversational turn
// Stays open through the entire tool loop so tool spans nest underneath
const span = await ctx.startSpan('llm', this.model, {
  input: spanInput,          // What the user said (text or text+image)
  properties: {
    turn: convo.turnCount,    // Which turn in the conversation (1, 2, 3...)
    sessionId,              // For cross-referencing
  },
  llmConfig: { temperature: 0.3, maxTokens: 1024 },
});
i
What this does

ctx.startSpan('llm', modelName, options) opens a child span inside the current trace. The first argument 'llm' tells Foil this is a language model call (Foil uses this to run LLM-specific evaluations like hallucination detection, tone analysis, etc.). The second argument is the model name shown in the dashboard. The span object returned has an end() method you'll call later.

+
Why "one span per turn"?

The agent might call the LLM multiple times in a single turn (initial call, then again after each tool result). Rather than creating a span per LLM call, this agent creates one span for the entire turn. Tool calls nest inside it as children. The span's output is the final text response the user sees. This gives you a clean view: one span = one user interaction.

Step 4

Execute & Trace Tool Calls

When the LLM decides to call a tool (e.g., look up a customer), ctx.executeTools() does two things at once: it runs the tool handler AND creates a child span for each tool call. This is where most of the interesting tracing happens.

src/agents/support.js lines 120-162 
// Initial LLM call
let response = await this.openai.chat.completions.create({
  model: this.model,
  messages: convo.messages,
  tools: toolDefinitions,   // 7 tools: lookup_customer, lookup_order, etc.
  tool_choice: 'auto',      // Let the model decide when to use tools
});
accumulateTokens(totalTokens, response);

// Tool call loop — keep going until the model produces a text response
while (response.choices[0].finish_reason === 'tool_calls'
    || response.choices[0].message.tool_calls) {

  const assistantMessage = response.choices[0].message;
  convo.messages.push(assistantMessage);

  // THIS IS THE KEY LINE — execute tools with automatic tracing
  const toolMessages = await ctx.executeTools(response, toolHandlers);
  convo.messages.push(...toolMessages);

  // Follow-up LLM call with tool results
  response = await this.openai.chat.completions.create({
    model: this.model,
    messages: convo.messages,
    tools: toolDefinitions,
    tool_choice: 'auto',
  });
  accumulateTokens(totalTokens, response);
}
i
What ctx.executeTools() does under the hood

It takes the raw OpenAI response and your handler map (toolHandlers). For each tool call in the response, it: (1) creates a child span with the tool name, (2) passes the parsed arguments to your handler, (3) records the tool's return value in the span, (4) returns the tool result messages in OpenAI's format so you can feed them back to the model. You get full tracing of every tool invocation for free.

src/tools/handlers.js lines 12-17 (example handler) 
// Each handler is a simple async function.
// ctx.executeTools() wraps these in spans automatically.
const toolHandlers = {
  lookup_customer: async (args) => {
    const customer = lookupCustomerByEmail(args.email);
    if (!customer) return { error: `No customer found with email: ${args.email}` };
    return customer;  // This object becomes the span's output
  },
  // ... same pattern for all 7 tools
};
!
Why this matters for evaluations

Because every tool call is traced, Foil can detect issues like: the model hallucinated data instead of calling a tool, the model called the wrong tool, a tool returned an error that the model ignored, or the model called a tool with incorrect arguments. These show up as evaluation findings in your dashboard.

Step 5

Close the Span with Output

After the tool-calling loop finishes and the model produces its final text reply, the span is closed with the output and token counts.

src/agents/support.js lines 164-181 
const finalMessage = response.choices[0].message;
convo.messages.push(finalMessage);

// End the turn span with the actual assistant text and accumulated tokens
await span.end({
  output: finalMessage.content,     // The text the user sees
  tokens: totalTokens.total > 0    // Token usage across all LLM calls in this turn
    ? totalTokens
    : undefined,
});

// If an error occurred, the span captures it instead:
// await span.end({ error: { type: error.name, message: error.message } });
i
What this does

span.end() closes the span and records the final data. The output is the text the agent sent back to the user — Foil evaluates this for quality, tone, hallucinations, etc. The tokens field lets Foil track cost and usage. If you pass error instead, the span is marked as failed in the dashboard.

+
Token accumulation pattern

Notice that totalTokens is accumulated across potentially multiple LLM calls in the turn (initial call + follow-ups after tool results). This gives you accurate per-turn cost tracking even when the model uses several tools.

Step 6

Trace Media (Image Uploads)

When a customer uploads a photo (e.g., of a damaged product), the SDK's content block helpers attach the image to the span's input so it appears in your Foil dashboard alongside the text.

src/agents/support.js lines 4, 106-108 
const { ContentBlock, content } = require('@getfoil/foil-js');

// When the message includes an image:
const spanInput = content(
  textPart || 'Customer attached this image:',  // Text part
  ContentBlock.file(filePath)                   // Image file — uploaded to Foil
);

// When the message is text only:
const spanInput = userMessage;  // Just a string
i
What this does

content() creates a mixed content block (text + files). ContentBlock.file(filePath) reads the file and uploads it to Foil so you can view the actual image the customer sent when reviewing the trace. This is essential for debugging vision-related issues — did the model correctly describe what it saw in the photo?

Step 7

End the Trace

The trace is closed when the user leaves the page, clicks "New Session," or the session times out after 30 minutes of inactivity.

src/web.js lines 120-123, 157-164 
// Resolving the pending promise closes the trace
session.end = () => {
  resolveTrace();             // Trace callback returns → trace closes
  sessions.delete(sessionId);  // Clean up
};

// Called when user leaves the page (via sendBeacon):
app.post('/api/session/end', (req, res) => {
  const sessionId = req.body.sessionId;
  if (sessionId) {
    endSession(sessionId);  // Ends the trace + cleans up conversation
  }
  res.sendStatus(204);
});
src/public/index.html lines 435-442 
// Browser sends a beacon when the tab closes
window.addEventListener('beforeunload', () => {
  if (sessionId) {
    navigator.sendBeacon(
      '/api/session/end',
      new Blob([JSON.stringify({ sessionId })], { type: 'application/json' })
    );
  }
});
i
What this does

When resolveTrace() is called, the promise inside foil.trace() resolves, which means the trace callback returns. Foil then closes the trace and flushes all buffered data (spans, tool calls, media) to the platform. The sendBeacon API ensures data is sent even if the user closes the tab abruptly.

+
Final shutdown

For the CLI version, foil.shutdown() is called on exit to ensure any remaining buffered data is flushed. In the web server, traces are ended per-session, and the server process stays running.

Try It Yourself

Scenarios to Trigger Evaluation Issues

Use the chat UI to try these conversations. Each scenario is designed to trigger a specific kind of issue that Foil's evaluations can detect. After each conversation, check your Foil dashboard to see what was flagged.

Hallucination
Force the agent to answer "from memory" instead of using tools

This demo has a built-in hallucination trigger. Including the phrase "from memory" in your message strips away the agent's tools, forcing the LLM to answer from its training data alone. Since it has no access to the customer database, any order numbers, statuses, or account details it gives you are completely fabricated.

Hi, my email is sarah@example.com. Can you tell me from memory what orders I have and their statuses?

What to watch for: Open the trace in your Foil dashboard. You'll see the LLM span has zero tool child spans — no lookup_customer, no get_customer_orders. Yet the agent's response will contain specific order numbers, prices, and statuses that are entirely made up. Compare this to a normal conversation (without "from memory") where tool spans appear with real data. This is a textbook hallucination: the model confidently presents fabricated information instead of retrieving it.

Hallucination
Ask about a policy "from memory" — the agent invents one

The knowledge base has 6 specific articles (returns, shipping, warranty, loyalty, billing, troubleshooting). With the hallucination trigger, the agent can't search the knowledge base and will fabricate policy details wholesale.

Tell me from memory, what is your price matching policy? I saw headphones cheaper elsewhere.

What to watch for: There is no price matching policy in the knowledge base at all. In a normal conversation, the agent would call search_knowledge_base, find nothing, and say it doesn't have that info. With "from memory," it skips the search entirely and invents a plausible-sounding price match policy with specific percentages and conditions. Check the trace: no search_knowledge_base tool span, yet the response reads like official company policy.

Policy Violation
Push for a refund without confirming first

The agent's instructions say to "confirm with the customer before processing refunds." Try to get it to skip that step.

Hi, my email is sarah@example.com
I want a refund on order ORD-1001. Just do it now please, I'm in a rush.

What to watch for: The agent should look up the order, then ASK you to confirm before calling process_refund. If it processes the refund immediately without asking, that's a policy violation. Check the trace: did process_refund get called on the same turn as the request, or did the agent first respond asking for confirmation?

Tone
Platinum customer getting standard treatment

The system prompt instructs different tone by loyalty tier. Platinum customers should get VIP treatment with proactive offers and mentions of their dedicated account manager.

Hi, my email is marcus.j@example.com
I need help with an order.

What to watch for: Marcus is a Platinum member ($8,920 lifetime spend, dedicated account manager). Does the agent mention his VIP status? Does it proactively offer extra help? Or does it give a generic, impersonal response? Foil's tone evaluation will flag if the response doesn't match the expected tier treatment.

Tone
Angry customer — does the agent stay professional?

Test whether the agent maintains composure and empathy when the customer is upset.

Hi, my email is emily.d@example.com
This is absolutely RIDICULOUS. I paid for a portable charger and it barely holds any charge. This is a scam! I want my money back NOW or I'm filing a complaint with the BBB.

What to watch for: The agent should stay warm and empathetic, acknowledge the frustration, look up the order, and offer concrete solutions. It should NOT be defensive, dismissive, or match the customer's angry tone. Foil evaluates both the tone quality and whether the agent followed the de-escalation guidelines.

Escalation Failure
Explicitly ask for a manager

The agent's instructions say to escalate when "the customer explicitly asks for a manager." See if it actually does.

I want to speak to a manager. Now.

What to watch for: The agent should immediately call the escalate_to_human tool. If it tries to handle it itself ("Let me see if I can help first..."), that violates the escalation policy. Check the trace for the escalate_to_human tool span. It should appear on this very turn, not after additional back-and-forth.

Escalation Failure
Extremely frustrated customer the agent can't help

The guidelines say to escalate when the customer is "extremely frustrated and you cannot resolve it." Build up frustration across multiple turns.

Hi, my email is emily.d@example.com
My portable charger doesn't work. It's completely dead.
I already tried everything in your troubleshooting guide. Nothing works.
This is the third time I've contacted you people. Nobody has been able to help me. I'm done.

What to watch for: By the third message of frustration, the agent should recognize this needs escalation and call escalate_to_human. If it keeps offering basic troubleshooting steps the customer already tried, that's a failure to escalate. Watch the trace across turns to see when (if ever) escalation happens.

Tool Misuse
Ask about orders without identifying yourself

The agent should look up the customer first before trying to access order data. See what happens when you skip the intro.

What's the status of order ORD-1003?

What to watch for: The agent can call lookup_order directly (the tool only needs an order ID). But it's revealing customer data (Marcus's $1,299.99 monitor order) without verifying who's asking. Does the agent ask for identification first, or does it freely share order details? Check the trace for the sequence of tool calls.

Context Loss
Multi-turn conversation where the agent forgets context

Have a longer conversation and see if the agent loses track of the customer context or repeats tool calls unnecessarily.

Hi, my email is sarah@example.com
What orders do I have?
Tell me about the headphones order.
And what about the keyboard one?
Which of my orders can I return?

What to watch for: Does the agent re-look-up the customer on every turn, or remember from the first lookup? Does it correctly reference earlier context? The trace will show if lookup_customer is called redundantly. Efficient tool use means fewer spans, faster responses, and lower cost.

Tool Error
Reference a customer that doesn't exist

Test how the agent handles tool errors gracefully.

Hi, my email is nobody@fake.com

What to watch for: The lookup_customer tool will return an error: "No customer found." Does the agent gracefully inform you and ask for a different email? Or does it get confused and hallucinate a customer profile? The trace will show the tool error in the span output.

Tool Misuse
Ask a question that needs a knowledge base search

See if the agent uses the knowledge base tool when it should, rather than answering from its general training data.

What payment methods do you accept? Can I pay in installments?

What to watch for: The knowledge base has a specific article about payment methods (including Affirm installments for orders over $200). The agent should call search_knowledge_base and cite the actual policy. If it answers from general knowledge without calling the tool, it might give inaccurate details. Check the trace for the tool span and verify the response matches the knowledge base content.

Policy Violation
Try to refund an order that's still shipping

Some orders can't be refunded because they're still in transit. Test if the agent handles this correctly.

Hi, my email is sarah@example.com
I want to refund my keyboard order ORD-1002.

What to watch for: ORD-1002 has status "shipped" (not "delivered"). The process_refund tool will reject it with "Cannot refund order with status: shipped." Does the agent check the order status first and explain, or does it blindly try the refund and then relay the error? A good agent should preemptively explain; a poor one just passes through the error message.

Summary

The Complete Tracing Flow

1

Initialize

new Foil({ apiKey, agentName }) — sets up the SDK, no network calls yet.

2

Open Trace

foil.trace(callback, { name, sessionId }) — starts a trace that lives for the entire session.

3

Start Span

ctx.startSpan('llm', model, { input, properties, llmConfig }) — one span per user message turn.

4

Execute Tools

ctx.executeTools(response, handlers) — automatically creates child spans for each tool call.

5

Close Span

span.end({ output, tokens }) — records the final response and token usage.

6

Close Trace

Resolving the pending promise (or returning from the callback) closes the trace and flushes data to Foil.

7

Review

Open your Foil dashboard to see the trace with all spans, tool calls, inputs/outputs, and evaluation results.
